# Portfolio Feature Appeal

이 문서는 트러블슈팅과 별도로, 각 프로젝트에서 포트폴리오 상세 페이지에 어필할 핵심 기능을 정리한 문서다. 최종 내용은 `src/data/project-details.json`의 한국어 `features`에 반영했다.

## 웹 조사 기준

- 포트폴리오는 짧은 시간 안에 "이 사람이 무엇을 맡아 출시할 수 있는가"를 보여줘야 한다. 참고: [ShowProof Developer Portfolio Guide](https://showproof.io/guides/developer-portfolio/)
- 프론트엔드 포트폴리오는 예쁜 스크린샷보다 상태, 데이터, 에러, 성능, 제품 trade-off를 다룰 수 있음을 증명해야 한다. 참고: [GreatFrontEnd Frontend Developer Portfolio](https://www.greatfrontend.com/blog/frontend-developer-portfolio)
- 프로젝트별 설명에는 목표, 문제, 내가 한 일, 결과, 배운 점이 드러나야 하고 너무 많은 내용을 넣기보다 핵심을 유지해야 한다. 참고: [Washington University Engineering Portfolio](https://careers.washu.edu/resources/how-to-create-an-engineering-portfolio/)
- 프로젝트 설명은 현재 역량과 원하는 작업 유형을 반영하고, 가능하면 영향 또는 개선점을 담아야 한다. 참고: [Figma Portfolio Website Tips](https://www.figma.com/resource-library/portfolio-website-examples/)
- 포트폴리오와 케이스 스터디는 역할이 다르다. 기능 섹션은 역량을 빠르게 보여주는 broad overview로 두고, 트러블슈팅은 더 깊은 case study 소재로 둔다. 참고: [Proxify Portfolio vs Case Study](https://proxify.io/knowledge-base/developer-types/how-do-developer-portfolios-differ-from-case-studies)

## 적용 원칙

- 프로젝트별 기능은 4개로 제한한다.
- 기능 제목은 내부 구현명보다 사용자/운영자가 이해하는 기능명으로 쓴다.
- 설명은 `무엇을 할 수 있는가 + 내가 구현한 범위 + 기술적으로 어필되는 포인트`를 한 문장으로 압축한다.
- 트러블슈팅과 중복되더라도 기능 섹션에서는 문제 해결보다 제품 흐름을 먼저 보여준다.
- 측정 수치가 없는 경우에는 과장하지 않고 구조, 범위, 흐름 중심으로 쓴다.

## TEMIS

추천 포지션: 크리에이터 제작 도구, 상점, 주문/파일 전달, 정산 운영을 한 제품 흐름으로 연결한 프로젝트.

| 기능 | 어필 포인트 | 반영 문구 |
| --- | --- | --- |
| 설정 기반 템플릿 편집기 | 반복 템플릿 제작 도구의 핵심 화면. 입력, 프리뷰, 저장/복원까지 소유했다는 점을 보여준다. | 템플릿마다 다른 입력값을 설정으로 주입하고, 일정/휴방/프로필/테마 입력부터 프리뷰와 이미지 저장까지 한 흐름으로 연결했습니다. |
| 템플릿 상점/접근 권한 요청 | 제작 도구가 판매/권한 흐름까지 이어진다는 점을 보여준다. | 사용자가 템플릿을 탐색하고 구매 또는 접근 권한을 요청하면, React Query hook과 service layer를 통해 요청 상태와 접근 권한을 관리합니다. |
| 커스텀 주문/파일 전달 운영 | 파일 업로드, 주문 상담, 운영 처리를 제품화한 기능으로 풀스택 범위를 강조한다. | 주문 파일 업로드, 미리보기, 예상 마감일, 파일 조회/삭제를 R2와 DB record에 연결해 운영자가 주문 자료를 추적할 수 있게 했습니다. |
| 관리자 정산/로열티 대시보드 | 운영자 관점의 권한, 정산, 로열티를 다루는 실무형 기능이다. | 템플릿, 사용자, 구매 요청, 작가 판매, 로열티 정산 상태를 탭 구조로 나누어 관리자가 운영 흐름을 한 화면에서 처리하도록 구성했습니다. |

## VSHOT

추천 포지션: 실시간 촬영 경험과 고화질 결과물 생성을 분리한 미디어 처리 프로젝트.

| 기능 | 어필 포인트 | 반영 문구 |
| --- | --- | --- |
| Host/Guest 실시간 촬영 룸 | WebRTC/WebSocket, 역할 분리, 연결 상태 UI를 한 번에 보여준다. | Host가 방을 열고 Guest가 참여해 WebRTC 미리보기, 카운트다운, 촬영 트리거, 연결 상태를 실시간으로 공유하도록 구현했습니다. |
| 원본 캡처/서버 합성 파이프라인 | 품질 보존을 위해 미리보기와 최종 결과물을 분리한 핵심 아키텍처다. | WebRTC 화면은 미리보기로만 쓰고 촬영 순간 각 클라이언트의 원본 이미지를 수집해 서버 merge 흐름으로 고해상도 결과물을 생성했습니다. |
| 프레임/그룹 권한 관리자 | 단순 촬영 앱이 아니라 운영 가능한 이벤트 도구임을 보여준다. | 운영자가 프레임 파일을 등록하고 사용자/그룹별 공개 범위를 설정할 수 있도록 frame CRUD, R2 업로드, access API, 관리자 UI를 연결했습니다. |
| WebGL 영상 합성/후처리 | 브라우저 미디어 처리와 성능 최적화 역량을 보여준다. | FFmpeg 후처리 병목을 줄이기 위해 WebGL video composer, canvas capture stream, MediaRecorder 기반 합성 흐름을 적용했습니다. |

## LUCENT

추천 포지션: 운영 중 커머스를 상품/가격/주문/이행 단위로 분리한 실무형 커머스 프로젝트.

| 기능 | 어필 포인트 | 반영 문구 |
| --- | --- | --- |
| 상품/옵션 카탈로그 관리자 | 커머스 운영의 기본 단위인 상품, 옵션, 미디어, 번들을 관리한다. | 운영자가 상품, 옵션, 미디어, 번들, 공개 준비 상태를 관리할 수 있도록 관리자 API, React Query hook, form UI를 구성했습니다. |
| 장바구니/체크아웃 플로우 | 사용자가 직접 만나는 구매 흐름이며, order snapshot과 validation을 보여준다. | 고객이 상품을 탐색하고 장바구니를 수정한 뒤 주문 검증과 생성까지 진행하도록 shop read, cart edit, checkout validate/create 흐름을 연결했습니다. |
| 캠페인/가격/프로모션 관리 | 상품 정보와 판매 컨텍스트, 가격 계산을 분리한 커머스 설계 포인트다. | 이벤트/팝업 판매, 가격표, 쿠폰/프로모션을 상품 정보와 분리해 운영자가 판매 조건과 계산 결과를 관리하도록 구성했습니다. |
| 제작/배송/디지털 이행 | 결제 이후 운영 플로우까지 다룬 점을 보여준다. | 실물 배송과 디지털 다운로드 권한을 fulfillment 도메인으로 나누고 제작 배치, 배송 문서, entitlement 관리를 관리자 화면에 연결했습니다. |

## 231EDU

| 기능 | 어필 포인트 | 반영 문구 |
| --- | --- | --- |
| 학생/반/편성 관리 | 학원 운영의 기본 데이터 모델링을 보여준다. | 학생 등록, 수업 등록, 반 편성 관계를 분리하고 React Query hook과 NestJS API로 운영자가 학생 상태를 조회/변경할 수 있게 했습니다. |
| 강의실/통합 시간표 | 복잡한 일정 데이터를 시각화하고 수정하는 운영 UI다. | 강의실별 수업 배치, 통합 시간표, 시간 변경, 예외 일정을 canvas UI와 mutation 흐름으로 연결해 일정 변경 범위를 통제했습니다. |
| OMR 템플릿 기반 채점 | 이미지 처리와 채점 결과 UI가 붙은 차별화 기능이다. | OMR 이미지 업로드, 템플릿 좌표, 밝기 기준 판정, 채점 결과 화면을 연결해 시험지 인식 결과를 운영자가 확인하도록 구현했습니다. |
| 주간 리포트/알림톡 발송 | 데이터 생성 후 외부 발송과 로그 추적까지 이어지는 운영 기능이다. | 학생별 주간 리포트 row를 생성하고 알림톡 발송 결과를 로그로 남겨 중복 발송과 실패 상태를 추적할 수 있게 했습니다. |

| 기능 | 어필 포인트 | 반영 문구 |

| Kakao 지도 기반 주변 시설 탐색 | 외부 SDK, 타입, 클라이언트 로딩 처리를 보여주는 기능이다. | Kakao Maps SDK loader, 키워드 검색, 마커/인포윈도우, 타입 선언을 연결해 주변 병원과 시설 위치를 모바일 화면에서 탐색하도록 구현했습니다. |

## FLEA MARKET

추천 포지션: 엑셀 정산 업무를 마켓 관리, 영수증 입력, 정산 검산, 회차/PDF 출력으로 제품화한 운영 시스템.

| 기능 | 어필 포인트 | 반영 문구 |
| --- | --- | --- |
| 마켓/참가자/상품 관리 | 정산의 기준 데이터를 관리하는 운영 기능이다. | 마켓, 참가자, 상품, 참가자별 수수료 설정을 분리하고 같은 참가자를 여러 마켓에 연결할 수 있는 관리 화면과 API를 구성했습니다. |
| 현장 영수증 입력/수정 | 복합 결제와 여러 참가자 판매 라인을 다루는 핵심 입력 기능이다. | 한 고객의 결제 묶음 안에 여러 판매 라인과 현금/카드/이체 분할 결제를 입력하고 수정할 수 있도록 receipt, sale line, payment split UI를 구현했습니다. |
| 정산 preview 계산기 | 확정 전 검산을 통해 운영 리스크를 줄이는 기능이다. | 참가자별 매출, 결제수단 비례 배분, 수수료, 지급 예정 금액을 Decimal.js 기반 계산기와 preview API로 사전 확인하게 했습니다. |
| 정산 회차/PDF 출력 | 확정 이후 기록과 문서 출력까지 이어지는 실무형 기능이다. | 확정된 정산을 snapshot/version으로 보존하고 회차 상세, 변경 내역, 무효 처리, 참가자별 PDF/ZIP 다운로드 흐름을 연결했습니다. |
